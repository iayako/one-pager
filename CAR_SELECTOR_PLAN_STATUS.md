# План и статус задачи: демо селектора авто

## Цель
Сделать демо-версию селектора автомобилей по логике, похожей на `auto.drom.ru`, с опорой на данные `vPIC` и собственной БД.

## Текущий статус
- Статус: **`demo / in progress`** — базовый UX и данные на месте; полный REST-справочник и прод-контур ещё не закрыты.
- Этап: демо интегрировано в одностраничный калькулятор; изображения подтягиваются из Wikipedia и хранятся локально.
- Последнее обновление: **2026-04-27**

## Что уже сделано

### Исследование и стратегия
- Проверен исходный API CarQuery: endpoint из примера недоступен (ошибка/404), как источник не подходит.
- Проверен `vPIC`:
  - Запрос моделей по марке и году работает.
  - Для `Toyota 2020` получен список моделей.
  - VIN decode (`DecodeVinValuesExtended`) возвращает расширенные характеристики.
- Принято рабочее направление: не ходить из браузера во внешнее API на каждый выбор; гибрид локальная БД + backend + точечные запросы.

### Реализовано в репозитории (демо)
- **SQLite:** файл `car_selector.db`, схема в `car_selector_schema.sql` — таблицы `vehicles` (в т.ч. VIN, make/model/year, поля из vPIC) и `vehicle_images` (`image_url`, `is_primary`, `source_name`).
- **Backend:** `GET api/vehicle_images.php` — отдаёт JSON списка авто с основной картинкой (LEFT JOIN на `vehicle_images`). Параметр `?refresh=1` дозаписывает **только отсутствующие** URL картинок через англоязычную Wikipedia API (логика как в коде PHP).
- **Offline / batch:** `scripts/refresh_vehicle_images.py` — для каждой машины подбирает изображение (те же кандидаты страниц + поиск), обновляет или создаёт строку в `vehicle_images`; при неудаче ставит SVG-заглушку; устойчивость к обрывам соединения (повторы HTTP).
- **Вспомогательно:** `scripts/run_vehicle_images_refresh.php` обёртка над API с `refresh=1`; другие скрипты в `scripts/` — сиды и альтернативные сценарии изображений (по марке, demo loremflickr и т.д.).
- **Frontend:** в `index.html` / `app.js` — блок выбора авто: быстрый поиск по пресетам, раскрываемый каскад марка → модель → год, сетка карточек с фото; данные загружаются с `api/vehicle_images.php`.

### Известные ограничения демо
- Не все марки/модели имеют подходящую страницу с `pageimage` в Wikipedia — часть записей остаётся с `source_name = placeholder`.
- Эндпоинты вида `GET /api/makes`, `/api/models` из первоначального MVP **ещё не вынесены** отдельно; каталог целиком приходит одним запросом.

## Предложенная архитектура (MVP) — целевая

1. Источник данных: справочник make/model/year из `vPIC` в свою БД.
2. Backend: отдельные REST endpoint для каскада (или пагинация).
3. Frontend: каскадный селектор + поиск.
4. Дополнительно: VIN через backend, кэш и retry для внешних вызовов.

Текущая реализация — упрощённый шаг к п.1–3 за счёт одного JSON-endpoint и SQLite.

## Риски и ограничения
- `vPIC` не закрывает все коммерческие характеристики; для глубины данных — VIN или второй источник.
- Wikipedia API и сеть: лимиты и нестабильность; для продакшена имеет смысл заранее прогонять `refresh_vehicle_images.py` и хранить только URL в БД.

## Решение по стратегии интеграции
- Подход: **локальная БД + свои API + точечные вызовы внешнего API**.
- Прямые запросы к `vPIC` из браузера в runtime не использовать как основной путь.
